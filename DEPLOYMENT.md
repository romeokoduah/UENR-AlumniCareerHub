# Deployment Guide

## Architecture

UENR Alumni Career Hub is a **split full-stack app** built to run entirely on free tiers:

- **Frontend** (`client/`) — React + Vite SPA, built to static assets. Hosted on any CDN/static host.
- **Backend** (`server/`) — Express + Prisma. Runs on any Node host. **Stateless** — no local filesystem dependencies in production.
- **Database** — Neon (managed Postgres with generous free tier and branching)
- **Image storage** — Cloudinary (free tier: 25 GB storage, 25 GB bandwidth)

Everything that used to require a persistent disk (SQLite DB file, uploaded landing photos, editable landing content) now lives in Neon or Cloudinary. The backend can be deployed to any free-tier Node host and you can redeploy all day without losing data.

## Recommended: Render free + Neon + Cloudinary (one-click)

This is exactly what `render.yaml` at the repo root configures. Total cost: **$0/month**.

### Prerequisites (~3 minutes)

1. **Neon account** — https://console.neon.tech
   - Create a project named anything (e.g. `uenr-career-hub`)
   - Copy the two connection strings from the dashboard:
     - **Pooled** (contains `-pooler`) → use for `DATABASE_URL`
     - **Direct** (no `-pooler`) → use for `DATABASE_URL_UNPOOLED` (Prisma needs this for schema operations)
2. **Cloudinary account** — https://cloudinary.com
   - From the dashboard's **"Product Environment Credentials"** section, grab the **API Environment variable** string — it looks like `cloudinary://123456789:AbCdEf@your-cloud-name`
3. **Anthropic API key** *(optional — only required for CareerMate chatbot, CV review, and mock interviewer)* — https://console.anthropic.com

### One-click deploy

Click the **"Deploy to Render"** button in `README.md` (or visit https://render.com/deploy?repo=https://github.com/romeokoduah/UENR-AlumniCareerHub directly).

Render reads `render.yaml` and creates two services:

- `uenr-career-hub-web` — static site, **free plan**
- `uenr-career-hub-api` — web service, **free plan**

When prompted for secret env vars (the ones marked `sync: false`), paste the values you collected above. That's `DATABASE_URL`, `DATABASE_URL_UNPOOLED`, `CLOUDINARY_URL`, and `ANTHROPIC_API_KEY`.

Everything else is auto-wired:
- `JWT_SECRET` — auto-generated by Render
- `CLIENT_ORIGIN` — pulled from the frontend service hostname
- `VITE_API_URL` — pulled from the backend service hostname

Click **Apply**. First build takes ~4 minutes (the backend runs `npm install`, `prisma generate`, `prisma db push`, and the idempotent seed script).

### Post-deploy check

1. Open the frontend URL (something like `https://uenr-career-hub-web.onrender.com`)
2. Log in with `admin@uenr.edu.gh` / `admin12345`
3. Go to **Admin → Landing page editor**
4. Upload a new photo. It should appear immediately, persist after refresh, and survive a redeploy.

If uploads fail with a 500: check that `CLOUDINARY_URL` is set correctly on the `uenr-career-hub-api` service.

If the opportunity board is empty: the seed didn't run. Check the build logs for the seed step — it's idempotent, so you can re-run it manually via **"Shell"** in the Render dashboard:
```bash
npx tsx prisma/seed.ts
```

## Free-tier limits to know about

- **Render free tier spins down** after 15 minutes of inactivity. The first request after idle takes ~30 seconds to wake. For a demo this is fine; for real traffic, upgrade to Starter ($7/mo) which stays always-on.
- **Neon free tier** gives you 0.5 GB storage, 100 compute hours/month, and auto-suspends after 5 minutes idle. Plenty for this app.
- **Cloudinary free tier** is 25 GB storage and 25 GB bandwidth/month. Well beyond what a demo alumni platform will use.

## Alternatives

### Vercel (frontend) + Fly.io (backend)

Same free-tier story, different hosts. Fly.io's free plan doesn't auto-sleep, so the API stays warm. Requires their CLI and a `fly.toml` — happy to add one if you want.

### Single-host everything on Render

Cheaper than splitting (one free web service instead of two) and faster because there's no cross-origin API round-trip. Change the backend to serve the built client:

```ts
// server/src/index.ts (at the bottom, before httpServer.listen)
import { fileURLToPath } from 'url';
const clientDist = path.resolve(__dirname, '../../client/dist');
app.use(express.static(clientDist));
app.get('*', (_req, res) => res.sendFile(path.join(clientDist, 'index.html')));
```

Then update `render.yaml` to have one web service with a build command that installs both workspaces and builds the client.

### Vercel (frontend + serverless API)

`vercel.json` + `api/index.ts` deploy the whole stack to Vercel: the Vite SPA as static assets, Express as a single serverless function at `/api/*`.

**Required env vars** (Settings → Environment Variables, set for Production + Preview):

| Variable | Where to get it |
| --- | --- |
| `DATABASE_URL` | Neon pooled connection string (`-pooler` host) |
| `DATABASE_URL_UNPOOLED` | Neon direct connection string (only needed locally for `db:migrate`) |
| `JWT_SECRET` | Any long random string |
| `ANTHROPIC_API_KEY` | https://console.anthropic.com (required for chatbot / CV review / mock interview) |
| `CLOUDINARY_URL` | Cloudinary dashboard (optional; uploads fall back to local disk without it, which won't persist on Vercel) |

**Schema push + seed do NOT run during Vercel deploys.** Run them once from your machine against the Neon database:

```bash
# with DATABASE_URL and DATABASE_URL_UNPOOLED in your local .env
npm run db:migrate   # pushes prisma/schema.prisma to Neon
npm run db:seed      # loads demo users, opportunities, etc.
```

Re-run `db:migrate` only when the schema changes.

### Vercel frontend only (static marketing page)

If you just want to show off the landing page without the full backend, Vercel works great. `vercel.json` is already configured at the repo root. Auth/opportunities/editor won't work without a deployed backend, but the landing page, typography, and animations will.

## What no longer works (compared to the old SQLite setup)

- **`npm run dev` with zero config** — you now need a Neon connection string in `.env` even for local dev. Offset: Neon branching lets you spin up a throwaway dev database in seconds.
- **`server/data/landing.json`** — gone. Landing content now lives in a Postgres `SiteContent` table, edited via the same admin UI.
- **`server/uploads/*`** — only used as a local fallback when `CLOUDINARY_URL` isn't set. In production, all uploads go straight to Cloudinary and the URLs returned by the upload endpoint are absolute `https://res.cloudinary.com/...` URLs.

## Rollback

If something goes wrong after deploy, Render keeps the previous build and you can roll back in one click from the service's **Deployments** tab. Neon branches let you restore data by creating a new branch from a historical timestamp (**Branches → Create from point-in-time**).
